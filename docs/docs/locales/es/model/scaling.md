# Escalamiento

El Modelo Soteria está diseñado para escalar — tanto dentro de una célula como a través de comunidades. El escalamiento es siempre **opcional** y **decidido localmente**.

## Escalamiento Dentro de una Célula

A medida que una célula crece, puede necesitar distribuir la carga de trabajo:

### Añadiendo Co-Guías

Cuando un guía está abrumado:

| Señal | Solución |
|-------|----------|
| Tiempos de respuesta lentos | Añadir co-guía para compartir carga |
| Punto único de fallo | Entrenar guía de respaldo |
| Riesgo de agotamiento | Rotar responsabilidades |

### Subdividiendo Segmentos

Cuando un segmento crece más allá de ~15 miembros, considera:

- **División geográfica**: Este/Oeste, Norte/Sur, por vecindario
- **División funcional**: Conductores vs. despachadores, clasificadores vs. distribuidores
- **División por tiempo**: Equipo diurno vs. equipo nocturno

Cada sub-segmento obtiene su propio grupo de Signal y líder.

### Coordinadores Regionales

Para células que cubren áreas grandes:

```

│                   CÉLULA DE CIUDAD                       │
│                                                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │    Norte    │  │   Centro    │  │    Sur      │      │
│  │   Región    │  │   Región    │  │   Región    │      │

│         │                │                │              │

│                          │                               │
```

Los coordinadores regionales manejan logística para su área mientras mantienen prácticas de seguridad de toda la célula.

## Cuándo NO Escalar

El escalamiento es opcional. Algunos contextos funcionan mejor permaneciendo pequeños:

| Situación | Recomendación |
|-----------|---------------|
| Pueblo pequeño/remoto | Una sola célula puede servir indefinidamente |
| Bajo nivel de actividad | No crear estructura que no necesitas |
| Red de confianza aún desarrollándose | Crecer confianza antes de crecer estructura |
| Presión externa aumentando | Consolidar en lugar de expandir |

**Más pequeño puede ser más seguro.** Más estructura significa más complejidad y más puntos potenciales de fallo.

## Generando Nuevas Células

Cuando una nueva comunidad quiere comenzar:

### Método del Grupo Semilla

1. **Identificar fundadores**: 3-5 personas de la nueva área, avalados por la red existente
2. **Entrenamiento**: Los fundadores reciben entrenamiento completo de nivel guía
3. **Independencia**: La nueva célula opera autónomamente desde el día uno
4. **Sin jerarquía**: La célula original no tiene autoridad sobre la nueva célula

### Compartiendo Conocimiento (Opcional)

Las células pueden elegir compartir:
- Información legal actualizada
- Mejoras de prácticas de seguridad
- Refinamientos de plantillas
- Lecciones generales aprendidas

Las células NO deben compartir:
- Listas de miembros
- Detalles operativos
- Canales de comunicación
- Especificidades de estructura interna

### Modelo de Enlace (Opcional)

Si las células quieren conexión continua:

```
┌──────────────┐          ┌──────────────┐
│   Célula A   │          │   Célula B   │
│              │          │              │
│  ┌────────┐  │          │  ┌────────┐  │
│  │Enlace  │◄─┼──────────┼─►│Enlace  │  │
│  └────────┘  │          │  └────────┘  │
└──────────────┘          └──────────────┘
```

- Una persona designada por célula para comunicación inter-celular
- Comparte solo información no operativa
- No puede comprometer a su célula a nada

## Principios de Escalamiento

1. **Escalar cuando sea necesario, no cuando sea posible** — No crear estructura por sí misma
2. **Decisiones locales** — Cada célula/segmento decide cuándo y cómo escalar
3. **Mantener compartimentación** — El escalamiento no debe romper límites de seguridad
4. **Documentar mientras creces** — Estructuras más grandes necesitan procesos más claros
5. **Autonomía sobre eficiencia** — Células independientes son más seguras que coordinadas
